# 워크플로우 가이드 커맨드

빠른 참조를 위한 Atlassian MCP 기반 완전한 개발 워크플로우를 표시하는 명령어입니다.

## 🔄 완전한 개발 워크플로우

이 가이드는 이 프로젝트의 모든 개발 작업에 대한 5단계 프로세스를 보여줍니다.

---

## 1️⃣ 티켓 시작 (`/start-ticket`)

**목적**: Jira 티켓 작업 초기화

**작업**:
- ✅ git 상태 및 GitHub 인증 확인
- 🎫 embed-project에서 Jira 티켓 조회 및 선택
- 📋 티켓 상세 정보 및 Acceptance Criteria 표시
- 🌿 feature 브랜치 생성: `feature/[JIRA-ID]-[설명]`

**결과**: 명확한 요구사항이 있는 준비된 개발 환경

---

## 2️⃣ 개발 단계

**프로세스**:
- 💻 Acceptance Criteria를 따라 기능 구현
- 📝 설명적인 메시지로 점진적으로 커밋
- 🎫 커밋에 Jira 티켓 ID 포함: `🎫 EMBED-XXX`

**커밋 형식**:
```bash
git commit -m "feat(scope): 설명

- 구현 세부사항 1
- 구현 세부사항 2

🎫 EMBED-XXX"
```

---

## 3️⃣ Acceptance 체크 (`/check-acceptance`)

**목적**: 모든 요구사항이 구현되었는지 확인

**작업**:
- 🔍 브랜치 이름에서 티켓 ID 추출
- 📋 Jira에서 Acceptance Criteria 조회
- ✅ 구현과 각 조건 검토
- 📊 완료 리포트 생성

**결정**:
- 모두 완료 → 테스트 진행
- 미완료 → 개발 계속

---

## 4️⃣ 기능 테스트 (`/test-feature`)

**목적**: PR 전 종합 테스트

**프론트엔드 테스트** (Playwright):
- 사용자 상호작용 플로우
- 폼 검증
- UI 렌더링
- 접근성
- 반응형 디자인

**백엔드 테스트** (API):
- REST 엔드포인트 호출
- 응답 검증
- 에러 핸들링
- 데이터 지속성

**결정**:
- 모두 통과 → PR 생성
- 실패 → 수정 및 재테스트

---

## 5️⃣ 티켓 완료 (`/finish-ticket`)

**목적**: 워크플로우 완료 및 병합

**작업**:
- 📤 feature 브랜치를 원격으로 푸시
- 📝 전체 세부사항이 포함된 Pull Request 생성
- 🔀 PR 병합 (squash 및 브랜치 삭제)
- 🔄 로컬 main 브랜치 업데이트
- 🎫 Jira 티켓을 "Done"으로 업데이트

**결과**: 병합된 코드, 닫힌 티켓, 깨끗한 작업공간

---

## 📊 워크플로우 다이어그램

```
┌─────────────────┐
│ /start-ticket   │ → Jira 티켓 가져오기, 브랜치 생성
└────────┬────────┘
         ↓
┌─────────────────┐
│  개발 단계       │ → 기능 구현, 점진적 커밋
└────────┬────────┘
         ↓
┌─────────────────┐
│ /check-acceptance│ → Acceptance Criteria 확인
└────────┬────────┘
         ↓
    모두 완료?
         ├─ 아니오 → 개발 계속
         ↓
       예
         ↓
┌─────────────────┐
│ /test-feature   │ → Playwright/API 테스트 실행
└────────┬────────┘
         ↓
   테스트 통과?
         ├─ 아니오 → 이슈 수정, 재테스트
         ↓
       예
         ↓
┌─────────────────┐
│ /finish-ticket  │ → PR, 병합, Jira 업데이트
└────────┬────────┘
         ↓
┌─────────────────┐
│ 다음 티켓       │ → /start-ticket
└─────────────────┘
```

---

## ✅ 모범 사례

**반드시 할 것**:
- ✅ 모든 세션을 `/start-ticket`으로 시작
- ✅ 브랜치 명명 규칙 준수: `feature/EMBED-XXX-설명`
- ✅ 커밋에 티켓 ID 포함: `🎫 EMBED-XXX`
- ✅ 테스트 전 Acceptance Criteria 확인
- ✅ 병합 전 모든 테스트 실행
- ✅ Jira 티켓 상태 업데이트

**절대 금지**:
- ❌ Jira 티켓 없이 작업
- ❌ Acceptance Criteria 체크 생략
- ❌ 테스트 통과 없이 병합
- ❌ main 브랜치에서 직접 작업
- ❌ 모호한 커밋 메시지 사용

## 🎯 빠른 명령어 참조

| 명령어 | 목적 | 사용 시기 |
|--------|------|-----------|
| `/start-ticket` | 새 티켓 시작 | 작업 시작 |
| `/check-acceptance` | 요구사항 확인 | 구현 후 |
| `/test-feature` | 모든 테스트 실행 | PR 생성 전 |
| `/finish-ticket` | 병합 및 종료 | 테스트 통과 후 |
| `/workflow` | 이 가이드 표시 | 언제든지 (참조용) |

## 💡 팁

1. **점진적 커밋**: 의미 있는 메시지로 자주 커밋하세요
2. **조기 테스트**: 끝까지 기다리지 말고 테스트하세요
3. **AC 주의 깊게 읽기**: Acceptance Criteria는 계약입니다
4. **질문하기**: 요구사항이 불명확하면 먼저 Jira에서 명확히 하세요
5. **조직적으로 유지**: 하나의 티켓, 하나의 브랜치, 하나의 집중
